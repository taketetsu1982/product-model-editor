# /generate

PRD・MRD等の入力からプロダクトモデルJSON（objects, views, paneGraph, screens）を一括生成する。
オブジェクト抽出にはOOUI（オブジェクト指向UI）モデリング手法を用いる。
入力がタスク記述（「〇〇する」「〇〇を行う」）で書かれていても、そこから対象物（名詞）を浮かび上がらせ、オブジェクトベースのモデルに変換する。

## 入力

PRD・MRDの内容、またはユーザーの説明からObjectとPaneを抽出してJSONを生成する。

## 出力

- `{EDITOR_DIR}/product-model.json` — 統合JSON

> **注:** `{EDITOR_DIR}` はプロジェクトごとに異なる。導入時に実際のパスに置換すること。

## 実行手順

### Step 1: 既存JSONを確認

`{EDITOR_DIR}/product-model.json` が存在する場合は読み込み、現状を把握する。

`_variants` キーが存在する場合（バリアント分岐中）:
- ユーザーに以下を確認する: 「バリアントが存在しますが、generateを実行するとすべてのバリアントが上書きされます。よろしいですか？」
- OKならそのまま上書き（`_variants` は削除され通常形式で出力される）
- キャンセルなら処理を中断

### Step 1.5: 入力の曖昧さを解消する

Step 2 に進む前に、入力を読んで以下の観点で曖昧な箇所がないか確認する。**曖昧な場合は推測せず、ユーザーに質問して解消する。**

| 曖昧さの種類 | 例 | 質問例 |
|---|---|---|
| オブジェクトの境界が不明 | 「ユーザー情報を管理」→ ユーザーとアカウントは別？ | 「ユーザーとアカウントは別のオブジェクトですか？」 |
| 親子関係の方向が不明 | 「チームとプロジェクトが関連」→ どちらが親？ | 「チームがプロジェクトを持つ（has-many）ですか？逆ですか？」 |
| 属性かオブジェクトか判断できない | 「カテゴリで分類」→ カテゴリはオブジェクト？属性？ | 「カテゴリは独立して作成・編集しますか？それともタスクの属性ですか？」 |
| 同一概念の重複 | 「メンバー」と「担当者」と「ユーザー」が混在 | 「メンバー・担当者・ユーザーは同じオブジェクトですか？」 |
| 対象デバイスが不明 | デバイスの言及がない | 「対象デバイスは mobile と desktop の両方ですか？」 |

**原則: 1つの質問にまとめて聞く。** 曖昧な箇所が複数ある場合も、1回のメッセージで全質問を提示する。回答を待ってから Step 2 に進む。

入力が十分に明確な場合は、このステップはスキップしてよい。

### Step 2: 名詞と動詞を分離する

入力（PRD・MRD・ユーザー説明）を読み、以下を行う:

1. 入力文中の **名詞**（対象物・概念）と **動詞**（操作・タスク）をすべて抜き出す
2. タスク記述（「〇〇を登録する」「〇〇を検索する」等）は **名詞部分と動詞部分に分離** する
   - 例:「案件を検索する」→ 名詞「案件」＋ 動詞「検索する」
   - 例:「レポートを月次で生成する」→ 名詞「レポート」＋ 動詞「生成する」
3. 抜き出した名詞を **オブジェクト候補リスト** 、動詞を **アクション候補リスト** として保持する

> **原則:** PRD/MRDはタスク記述で書かれていることが多い。「動詞→名詞」のタスク順序を「名詞→動詞」のオブジェクト順序に反転させるのがこのステップの目的。

### Step 3: オブジェクトを判定する

オブジェクト候補リストの各名詞に対し、以下の **2条件** で判定する:

**条件A: 複数インスタンス性**
- 同じ属性（プロパティ）を持った複数のものが存在しうるか？
- 「案件」→ 案件A, 案件B, 案件C… → ✅ オブジェクト
- 「ログイン」→ 複数の同種インスタンスは想定しにくい → ❌

**条件B: 共通アクション性**
- それらに対して共通の処理（アクション）を行えるか？
- 「案件」→ どの案件にも「編集」「削除」「ステータス変更」ができる → ✅
- 「フラグ」→ フラグ自体に処理を行うのではなく、別オブジェクトの属性 → ❌

**排除すべきもの:**
- **属性（プロパティ）に過ぎないもの** — 「ステータス」「フラグ」「カテゴリ」等。これらはオブジェクトのfieldsになる
- **タスクが名詞化しただけのもの** — 「受信」「承認」「集計」等。動詞的にも名詞的にも取れるものはアクション候補に戻す
- **UIコンポーネント名** — 「ダッシュボード」「設定画面」等。これらはPaneの表現であってObjectではない

> 📖 [patterns.md §1「よくある間違いパターン」](patterns.md) の M1〜M5 で典型的な誤判定を確認すること。
> 📖 [examples.md「Good/Bad具体例」](examples.md) で入力→出力の変換イメージを掴むこと。

判定を通過したものを **objects** として確定し、リレーションを定義する。

**リレーション定義のルール:**
- **一方向のみ**: 親→子の方向でのみ定義する。A→Bを定義したらB→Aは定義しない
- **重複禁止**: 同じオブジェクト間に同じtypeのリレーションを複数定義しない

### Step 4: Paneを導出する

確定した各オブジェクトに対し、Paneを検討する:

**原則: collection + single を基本とするが、機械的に両方作らない。**

各オブジェクトについて以下を問う:

1. **collectionは必要か？**
   - ユーザーが複数インスタンスを一覧・比較・選択する場面があるか？
   - あれば collection を作り、一覧に表示すべき主要属性を fields に設定する

2. **singleは必要か？**
   - そのオブジェクトの詳細情報をユーザーが閲覧・編集する場面があるか？
   - グルーピングの手がかりに過ぎないオブジェクト（例: メールボックス、フォルダ）は single を省略またはオプション扱いにできる

3. **verbsの配置**
   - Step 2で分離した動詞（アクション候補）を、対応するオブジェクトのPaneに割り当てる
   - アクションは **独立したメニュー項目にしない** 。必ずオブジェクトのPane上に配置する
   - collection と single で使えるアクションが異なる場合は、それぞれのPaneに適切に振り分ける

4. **追加Paneの検討**
   - 同一オブジェクトに対して、異なる切り口のPaneが必要な場合がある（例: カレンダー表示とリスト表示）
   - その場合は同じ objectId を持つ別Paneとして定義し、prompt にPane固有の表現意図を記述する

5. **promptの記述ルール**
   - そのPane自身が提供する情報と操作のみ記述する
   - 他Paneとの画面構成（「サイドパネルに○○を表示」等）は書かない。Paneの組み合わせはデバイスごとに異なるため

> 📖 [patterns.md §3「fields選び方ガイド」](patterns.md) で collection/single の fields 選定基準を確認すること。

### Step 5: Pane Graphを定義する

Pane間の関係をグラフの辺として定義する。`from`/`to` にはPane IDのみ指定する。辺には2種類ある:

**drilldown（有向辺・矢印）**: 同一ObjectのCollection → Singleへの遷移
- **collection → single** が基本パターン（一覧から1件選んで詳細へ）
- param には遷移先へ渡すパラメータを日本語で記述する（例: 「プロジェクトID」）
- 異なるObject間のdrilldownは原則定義しない

**embed（無向辺・線）**: 上位ObjectのSingleが下位ObjectのCollectionを内包する親子関係
- 例: プロジェクト Single — タスク Collection
- Object間のhas-many/has-oneリレーションに対応する

- Pane GraphはScreenのグルーピングとは独立。どのPaneがどのPaneに関連するかのみを定義する

> 📖 [patterns.md §2「Pane Graph判定フローチャート」](patterns.md) で辺の種類を判定すること。

### Step 6: DevicesとScreenを定義する

対応デバイスを確認する（デフォルト: mobile と desktop）。`devices` フィールドの出力は不要（エディタが自動補完する）。
デバイスごとにPaneをグルーピングしてScreenを定義する:

- 同じ論理画面（例: 「ホーム」）をデバイスごとに定義する
- **mobile**: 1つのPaneで構成するのが基本。画面遷移で別Paneに移動する
- **desktop**: 複数のPaneを組み合わせて1画面にする（例: 一覧 + 詳細のList-Detail）
- embed関係にあるPaneは、desktopでは同じScreenに配置し、mobileでは別Screenに分離するのが典型パターン

> 📖 [patterns.md §4「Screen構成のデバイス別パターン集」](patterns.md) でP1〜P5のパターンと判定手順を確認すること。

既存JSONがある場合は、Step 2〜6 を通じて既存の内容をベースに差分更新する。

### Step 7: JSONファイルを書き出す

`{EDITOR_DIR}/product-model.json` にJSONを書き出す。

### Step 8: 完了メッセージ

```
Product Model generated:
- JSON: {EDITOR_DIR}/product-model.json

エディタで確認・編集する場合は /edit を実行してください。
```
