# /model

PRD・MRD等の入力からプロダクトモデルJSON（objects, actors, views, transitions）を一括生成する。
オブジェクト抽出にはOOUI（オブジェクト指向UI）モデリング手法を用いる。
入力がタスク記述（「〇〇する」「〇〇を行う」）で書かれていても、そこから対象物（名詞）を浮かび上がらせ、オブジェクトベースのモデルに変換する。

## Input

PRD・MRDの内容、またはユーザーの説明からオブジェクトとビューを抽出してJSONを生成する。

## Output

- `{EDITOR_DIR}/product-model.json` — 統合JSON

> **注:** `{EDITOR_DIR}` はプロジェクトごとに異なる。導入時に実際のパスに置換すること。

## JSON Schema

```json
{
  "objects": [],
  "actors": [],
  "views": [],
  "transitions": []
}
```

| フィールド | 説明 |
|---|---|
| objects | オブジェクト定義 |
| actors | アクター定義（ロール別のアクセス可能オブジェクト） |
| views | ビュー定義（type: "collection" / "single"） |
| transitions | ビュー遷移定義 |

### objects

```json
{
  "id": "kebab-case識別子",
  "name": "表示名",
  "relations": [
    { "id": "一意ID", "targetId": "対象オブジェクトid", "type": "has-many | has-one | many-to-many" }
  ]
}
```

- オブジェクトはフラットな配列。グルーピングはしない
- relationsの `type` は `has-many` / `has-one` / `many-to-many`

### actors

```json
{
  "id": "kebab-case識別子",
  "name": "アクター名",
  "objectIds": ["アクセス可能なobject id"]
}
```

- プロダクトを利用するロール（Owner, Member等）を定義
- `objectIds` でそのアクターがアクセスできるオブジェクトを指定

### views

```json
{
  "id": "kebab-case識別子",
  "name": "ビュー名",
  "type": "collection | single",
  "objectId": "object id",
  "fields": ["フィールド名"],
  "verbs": ["操作名"],
  "x": 60, "y": 60,
  "prompt": "このビューの実装についての補足指示（自然言語）"
}
```

- `type`: `collection` = 一覧表示、`single` = 単体表示
- `objectId`: このビューが表示するオブジェクトのid
- 1つのオブジェクトを複数のビューから参照できる

### transitions

```json
{ "id": "一意ID", "from": "view id", "to": "view id", "trigger": "遷移トリガー" }
```

## Execution Steps

### Step 1: 既存JSONを確認

`{EDITOR_DIR}/product-model.json` が存在する場合は読み込み、現状を把握する。

### Step 2: 名詞と動詞を分離する

入力（PRD・MRD・ユーザー説明）を読み、以下を行う:

1. 入力文中の **名詞**（対象物・概念）と **動詞**（操作・タスク）をすべて抜き出す
2. タスク記述（「〇〇を登録する」「〇〇を検索する」等）は **名詞部分と動詞部分に分離** する
   - 例:「案件を検索する」→ 名詞「案件」＋ 動詞「検索する」
   - 例:「レポートを月次で生成する」→ 名詞「レポート」＋ 動詞「生成する」
3. 抜き出した名詞を **オブジェクト候補リスト** 、動詞を **アクション候補リスト** として保持する

> **原則:** PRD/MRDはタスク記述で書かれていることが多い。「動詞→名詞」のタスク順序を「名詞→動詞」のオブジェクト順序に反転させるのがこのステップの目的。

### Step 3: オブジェクトを判定する

オブジェクト候補リストの各名詞に対し、以下の **2条件** で判定する:

**条件A: 複数インスタンス性**
- 同じ属性（プロパティ）を持った複数のものが存在しうるか？
- 「案件」→ 案件A, 案件B, 案件C… → ✅ オブジェクト
- 「ログイン」→ 複数の同種インスタンスは想定しにくい → ❌

**条件B: 共通アクション性**
- それらに対して共通の処理（アクション）を行えるか？
- 「案件」→ どの案件にも「編集」「削除」「ステータス変更」ができる → ✅
- 「フラグ」→ フラグ自体に処理を行うのではなく、別オブジェクトの属性 → ❌

**排除すべきもの:**
- **属性（プロパティ）に過ぎないもの** — 「ステータス」「フラグ」「カテゴリ」等。これらはオブジェクトのfieldsになる
- **タスクが名詞化しただけのもの** — 「受信」「承認」「集計」等。動詞的にも名詞的にも取れるものはアクション候補に戻す
- **UIコンポーネント名** — 「ダッシュボード」「設定画面」等。これらはビューの表現であってオブジェクトではない

判定を通過したものを **objects** として確定し、リレーションを定義する。

### Step 4: ビューを導出する

確定した各オブジェクトに対し、ビューを検討する:

**原則: collection + single を基本とするが、機械的に両方作らない。**

各オブジェクトについて以下を問う:

1. **collectionは必要か？**
   - ユーザーが複数インスタンスを一覧・比較・選択する場面があるか？
   - あれば collection を作り、一覧に表示すべき主要属性を fields に設定する

2. **singleは必要か？**
   - そのオブジェクトの詳細情報をユーザーが閲覧・編集する場面があるか？
   - グルーピングの手がかりに過ぎないオブジェクト（例: メールボックス、フォルダ）は single を省略またはオプション扱いにできる

3. **verbsの配置**
   - Step 2で分離した動詞（アクション候補）を、対応するオブジェクトのビューに割り当てる
   - アクションは **独立したメニュー項目にしない** 。必ずオブジェクトのビュー上に配置する
   - collection と single で使えるアクションが異なる場合は、それぞれのビューに適切に振り分ける

4. **追加ビューの検討**
   - 同一オブジェクトに対して、異なる切り口のビューが必要な場合がある（例: カレンダー表示とリスト表示）
   - その場合は同じ objectId を持つ別ビューとして定義し、prompt にビュー固有の表現意図を記述する

### Step 5: 遷移を定義する

ビュー間の遷移を定義する:

- **collection → single** が最も基本的な遷移（一覧から1件選んで詳細へ）
- 親オブジェクトのsingleから、子オブジェクトのcollectionへの遷移（例: 企業詳細 → 所属メンバー一覧）
- 遷移の trigger には、ユーザーの具体的な操作を記述する（「行をクリック」「＋ボタン」等）

### Step 6: アクターを定義する

ロール別のアクターを定義し、アクセス可能なオブジェクトを割り当てる。

既存JSONがある場合は、Step 2〜6 を通じて既存の内容をベースに差分更新する。

### Step 7: JSONファイルを書き出す

`{EDITOR_DIR}/product-model.json` にJSONを書き出す。

### Step 8: 完了メッセージ

```
Product Model generated:
- JSON: {EDITOR_DIR}/product-model.json

エディタで確認・編集する場合は /editor を実行してください。
```
